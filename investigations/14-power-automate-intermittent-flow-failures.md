# [問い合わせ] Power Automate のフローが断続的に失敗するようになった

## 問い合わせ内容

SharePoint トリガー → 承認 → メール送信の自動化クラウドフローで、2026-06-15 頃から断続的な失敗が増加。

- 同じフローでも成功回と失敗回が混在
- 失敗ステップが毎回固定されない（SharePoint 取得やメール送信など）
- 直近で大きな設定変更は認識していない

## 調査プロセス

1. 既存ナレッジを確認（Power Automate の断続失敗に直接対応する既存記事は未登録）。
2. Microsoft Learn で以下の観点を確認。
   - 実行履歴からの失敗ラン特定手順
   - 認証失敗（401/403）と一時失敗（500/502）の扱い
   - Retry policy / Run after を使った耐障害化
   - Power Automate と SharePoint のスロットリングに関する公式説明
3. 事実と未確認事項を分離し、原因を断定せずに「候補 + 判定手順」を作成。

## 調査結果

- **確認できた事実**:
  - Power Automate は実行履歴で失敗アクションの詳細確認が可能。
  - 一時的障害への対策として Retry policy と Run after が推奨される。
  - SharePoint Online は高負荷時に `429/503` を返し、`Retry-After` を使った再試行が必要。
  - フローが継続的にスロットリングされると停止対象になり得る。
- **未確認（Issue 情報だけでは確定不可）**:
  - 実際の失敗ランのステータスコード内訳（`429/503` 優勢か、`401/403` 優勢か）
  - 特定時間帯への失敗偏り、並列度や実行量との相関
  - 接続再認証や権限変更の有無

以上より、原因は単一断定せず、(1) 一時的スロットリング/負荷、(2) 認証状態の揺れ、(3) 並列実行過多を優先候補として切り分ける方針とした。

## 回答内容

Issue #14 には以下を回答方針として提示する。

- 断定を避け、確認済み事実と未確認事項を分離
- 原因候補を優先順で提示
- 「どの結果ならどの原因か」を対応づけた切り分け手順
  - `429/503` 多発 → スロットリング/一時負荷
  - `401/403` 多発 → 接続・認証/認可
  - `400/404` 多発 → 設定値や入力データ不整合
- Retry policy / Run after / 並列度見直しの実施手順
- 参照リンクとして Microsoft Learn を添付
