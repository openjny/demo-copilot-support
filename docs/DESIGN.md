# 設計思想 — なぜこのリポジトリが存在するか

## 背景・課題

IT サポートチームの日常業務には、以下のような構造的課題がある：

1. **暗黙知の属人化** — ベテラン担当者の頭の中にだけノウハウがあり、チームで共有されない
2. **同じ調査の繰り返し** — 似た問い合わせが来るたびに、ゼロから公式ドキュメントを調べ直す
3. **ナレッジの散逸** — メール、チャット、口頭で回答が完結し、再利用可能な形で残らない
4. **調査プロセスの不透明さ** — どう調べてその結論に至ったのかが後から追えない

## このリポジトリの存在理由

このリポジトリは **GitHub Copilot Coding Agent によるIT問い合わせ対応自動化** のデモ環境として存在する。

具体的には、以下を実証する：

- GitHub Issue として構造化された問い合わせを受け付ける
- Copilot Coding Agent が Microsoft Learn MCP を使い公式ドキュメントを自動調査する
- 調査結果を Git 管理されたナレッジとして蓄積する
- 問い合わせ元の Issue に回答をコメントする

これは「GitHub Copilot はコード生成だけのツールではなく、業務プロセスの自動化にも適用できる」ことを示すデモである。

## 設計思想

### 1. Issue = 問い合わせの入口

問い合わせを GitHub Issue テンプレートで構造化することで：
- カテゴリ、対象サービス、環境情報が定型フォーマットで収集される
- Coding Agent が解析しやすい入力が保証される
- Issue のライフサイクル（open → assigned → resolved → closed）が対応状況の可視化になる

### 2. Git 管理によるナレッジの永続化

ナレッジを Markdown + Git で管理することで：
- **履歴**: いつ・誰が・何を追記/修正したかが追跡可能
- **差分**: 技術情報のアップデートが diff で可視化される
- **レビュー**: Pull Request を通じたナレッジの品質担保が可能
- **検索**: ファイルベースなので全文検索が容易

### 3. 調査プロセスの記録

`investigations/` に Issue 固有の調査過程を残すことで：
- 「なぜその結論に至ったか」のトレーサビリティを確保
- 同種の問い合わせが来た際の参考情報として機能
- 担当者の交代時にも文脈が引き継がれる

### 4. 再利用可能な知識と個別調査の分離

| ディレクトリ | 性質 | 寿命 |
|---|---|---|
| `knowledge/` | 汎用的・再利用可能な技術ナレッジ | 継続的に更新 |
| `investigations/` | 特定 Issue に紐づく調査記録 | 対応完了後は凍結 |

この分離により、ナレッジの肥大化を防ぎつつ、個別事案の詳細も失わない。

## なぜ Git / GitHub か

| 要件 | Git/GitHub が提供する解 |
|---|---|
| バージョン管理 | コミット履歴で変更追跡 |
| コラボレーション | PR レビュー、Issue コメント |
| 自動化との親和性 | Copilot Coding Agent がネイティブに統合 |
| 検索性 | GitHub Code Search、ローカル grep |
| 可搬性 | プレーンテキスト(Markdown)なのでロックインなし |

## デモとしての位置づけ

このリポジトリは、以下のメッセージを伝えるために存在する：

> **GitHub Copilot は「コードを書く AI」ではなく「開発者のワークフローを支援する AI」である。**
> IT サポートのような非開発業務であっても、Issue・Git・Markdown という
> 開発者にとって馴染みのあるプリミティブの上に載せることで、
> Copilot の力を最大限に活用できる。

### デモで見せたいポイント

1. **MCP による公式ドキュメント参照** — Agent が Microsoft Learn を検索・取得し、正確な情報に基づいて回答する
2. **ナレッジの自動生成** — 調査結果がそのまま再利用可能なドキュメントとして残る
3. **Issue ドリブンのワークフロー** — 問い合わせから回答までが GitHub 上で完結する
4. **蓄積による改善** — 過去のナレッジが存在する場合、Agent がそれを参照して即座に回答する
